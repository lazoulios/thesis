\chapter{Background}
\label{ch:background}

In this chapter we will provide the necessary background and graph theory concepts needed for the rest of the thesis.
We will also provide a brief overview of classical algorithms for the solution of the Shortest Path Problem
(e.g. Dijkstra's\cite{dijkstraNoteTwoProblems1959}, A*\cite{hartFormalBasisHeuristic1968b}, ALT\cite{goldbergComputingShortestPath2005}, Contraction Hierarchies\cite{geisbergerContractionHierarchiesFaster2008}),
before going into newer algorithms developed in the area of \textit{Algorithms with Predictions} (or \textit{Learning Augmented Algorithms}).

\section{Graph Theory Basics}
We will start by defining some basic graph theory concepts that will be used throughout the thesis.
For the purposes of this thesis, we will use the standard notations as described in Cormen et al.\cite{cormenIntroductionAlgorithms2009}
\footnote{Definitions and notation follow Cormen et al.~\cite{cormenIntroductionAlgorithms2009}, specifically Appendix B.4 (Graph Theory)}

%definitions below are based on Cormen et al. Introduction to Algorithms (2009). if i used a wrong definition please let me know :) 
A \textit{directed graph} $G$ is defined as an ordered pair $G=(V, E)$ where $V$ is a non-empty set of vertices (or nodes/points) and $E$ is a set of edges (or arcs/lines) that connect pairs of vertices.
We will denote the number of vertices as $n = |V|$ and the number of edges as $m = |E|$.

A \textit{weighted graph} is a graph $G$ where each edge has a numerical value or \textit{weight} (or \textit{cost}) that represents the cost of traversing that edge.
For a given edge $e \in E$, we will denote its weight as $w(e)$.
We define a weight function $w: E \rightarrow \mathbb{R}_{\geq 0}$
that assigns a non-negative
\footnote{The assumption of non-negative weights is necessary for the correctness of Dijkstra's algorithm, which we will revisit on Section 2.3.}
real number to each edge in the graph.

A \textit{path} $P$ is defined as a sequence of vertices $P = (v_1, v_2, ..., v_k)$ from a source vertex $s$ to a target vertex $t$ 
such that for $v_1 = s$, $v_k = t$, and $(v_i, v_{i+1}) \in E$ for all $1 \leq i < k$.
The \textit{cost} (or \textit{length}) of a path $P$ is defined as the sum of the weights of the edges in the path, i.e.

\begin{equation}
    w(P) = \sum_{i=1}^{k-1} w(v_i, v_{i+1})
\end{equation}

We denote \textit{Shortest Path Distance} from $s$ to $t$ as $\delta(s, t)$, which is the minimum cost from $s$ to $t$ compared to all possible paths from $s$ to $t$. 
If no path exists from $s$ to $t$, we define $\delta(s, t) = \infty$.

\section{The Shortest Path Problem}
The Shortest Path Problem is one of the most studied problems in graph theory.
The problem focuses on finding the minimum cost from a starting vertex $s$ to a every other vertex $v \in V$
In other words finding the \textit{Shortest Path Distance} from $s$ to all $v \in V$ similarly to how we previously mentioned ($\delta(s,v)$), and construct the corresponding paths that achieve this cost.

This problem is used in technologies we use daily such as in GPS navigation, network routing, logistic networks and generally in any path finding need.
While many simple cases can be solved easily just by intuition, in today's era of Big Data we require the use of algorithms (like \textsc{DIJKSTRA}) to solve more complex and interconnected cases.

\subsection{SPP Variations}
There are many variations of the Shortest Path Problem that were analyzed and studied in order to satisfy the different needs of each use case.

The \textit{All-Pair Shortest Path} (APSP) Problem for example searches for the shortest path between every possible pair of nodes ($u , v$)in a graph.
A famous algorithm for the solution of this problem was designed by Robert W. Floyd\cite{floydAlgorithm97Shortest1962} in 1962 with a time complexity of $O(n^3)$.
Further more, in the cases of sparse graphs an algorithm was designed by Donald B. Johnson\cite{johnsonEfficientAlgorithmsShortest} in 1977 further reducing the time complexity to $O(n^2\log n+nm)$
(Although the algorithm performs the same of slightly worse in the case of dense graphs).

The \textit{Single-Destination Shortest Path} (SDSP) Problem searches for the shortest path from every node $v \in V$ to a single target node $t$. 
In this case one can solve the problem simply by reversing the order of the graph edges and running Dijkstra's algorithm\cite{dijkstraNoteTwoProblems1959} (which we will develop on Section 2.3).
It is important to note that in the case of an unordered graph the solution for the Single-Destination Shortest Path is the same as the solution of the Single-Source Shortest Path.

Lastly, another notable variation to mention is the \textit{k-Shortest Path} (kSP) Problem which instead of searching for only the shortest possible path, it searches for the k-shortest possible paths. 
This variation is commonly used in navigation systems offering user an alternative route to take besides the shortest one possible. 
A solution for this problem was designed by Jin Y. Yen\cite{yenFindingShortestLoopless1971a} in 1971 by executing Dijkstra's algorithms $O(n)$ times for each of the k shortest paths with a time complexity of $O(kn(E+n\log n))$.

\subsection{Single-Source Many-Targets Shortest Path}
In this thesis, we focus specifically on the \textit{Single-Source Many-Targets Shortest Path} (SSMTSP) problem, which is the version of the Shortest Path Problem addressed by the algorithm of Feijen and Schäfer~\cite{feijenDijkstrasAlgorithmPredictions2023}.

SSMTSP is a variation of the Single-Source Shortest Path (SSSP) problem, where instead of finding the shortest path from a source node to all other nodes in the graph, we search to find the shortest paths from a single source node to a target node $t \in T$, where $T$ is a set of specific target nodes.
More specifically:

Given a directed, weighted graph $G=(V, E)$, with a weight function 
\footnote{As mentioned before the assumption of $\mathbb{R}_{\geq 0}$ is necessary for Dijkstra's algorithm}
$w: E \rightarrow \mathbb{R}_{\geq 0}$
, a source node $s$, and a set of target nodes $T \subseteq V$, the goal is to find the shortest paths $\delta(s,t)$ from $s$ to $\exists$ $t \in T$.

This specific version of the problem is used in real-world scenarios such as in Points of Interest (POI) search in navigation systems, where a user in a specific location (source node) may want to find multiples options (target nodes) for a specific destination (e.g. gas stations, restaurants, etc.) and not just the closest one.
It is also used as a subroutine in logistics and specifically in the Last-Mile Delivery problem, where a distribution center (source node) needs to deliver goods to customers
\footnote{Usually the delivery routine is calculated with a variation of the \textit{Traveling Salesperson Problem} (TSP), but often the first step is to calculate the shortests paths by solving the SSMTSP problem}
(target nodes) in the most efficient way possible.
Furthermore, it is used in Content Delivery Networks (CDNs) where a server (source node) needs to deliver content to multiple users (target nodes) in the most efficient way possible. 

\section{DIJKSTRA Algorithm}
The most famous and widely used algorithm for the SSSP Problem and specifically for our case the SSMTSP Problem is the \textsc{DIJKSTRA} Algorithm\cite{dijkstraNoteTwoProblems1959} which was proposed by Edsger W. Dijkstra in 1959. 

\textsc{DIJKSTRA} works only in cases where the edges have a non-negative weight ($w(e) \ge 0$ for each $e \in E$). 

\subsection{Algorithmic Analysis}
\textsc{DIJKSTRA}\footnote{Definitions and notation follow Cormen et al.~\cite[Ch.~24.3]{cormenIntroductionAlgorithms2009}, specifically Chapter 24.3 (Dijkstra's Algorithm)} works with a set S of visited (or settled) nodes. 
At the start $S$ is an empty set, the distance to the source node $s$ is 0 and the distance to the rest of the nodes in the graph is $\infty$.
We repeat a process of taking the unvisited (or unsettled) node $u \in V-S$ with the minimum distance, adding $u$ to $S$, and \textit{relaxing} all edges leaving $u$. 

For a given edge $(u, v) \in E$ with weight $w(u, v)$, if the current known distance to $v$ is greater than the distance to $u$ plus the weight of the edge, 
the algorithm updates (\textit{relaxes}) the distance to $v$:

\begin{equation}
\text{If } dist(v) > dist(u) + w(u, v) \text{, then update } dist(v) = dist(u) + w(u, v)
\end{equation}

The exact code implementation of \textsc{DIJKSTRA} we will use is as follows:
\begin{algorithm}[H]
\caption{DIJKSTRA (G, w, s, T)}
$d(s) = 0$, $d(v) = \infty$ for all $v \in V \setminus \{s\}$ \tcp*{tentative distances}
PQ.INSERT$(s, d(s))$\;
\While{not PQ.EMPTY()}{
    $u =$ PQ.REMOVE-MIN() \tcp*{u becomes settled}
    \If{$u \in T$}{
        \textbf{STOP} \tcp*{stop when u is target}
    }
    \For{$(u,v) \in E$}{
        $tent = d(u) + w(u,v)$ \tcp*{tentative distance of v}
        RELAX$(u, v, tent)$\;
    }
}
\end{algorithm}

\begin{algorithm}[H]
\caption{RELAX(u, v, tent)}
\If{$d(v) > tent$}{
    \If{$d(v) == \infty$}{
        PQ.INSERT$(v, tent)$ \tcp*{add v to PQ}
    }\Else{
        PQ.DECREASE-PRIO$(v, tent)$ \tcp*{decrease priority}
    }
    $d(v) = tent$ \tcp*{update distance of v}
}
\end{algorithm}

\textsc{DIJKSTRA} is a greedy algorithm\footnote{\textit{Greedy} algorithms work by making the best choice in any specific step/iteration.} 
since it selects the "lightest" vertex thus selecting a local optimal choice, but unlike most greedy algorithms it guarantees the global optimal result everytime.

As mentioned in the introduction, through the use of \textit{Fibonacci Heaps} as proposed by Fredman and Tarjan\cite{fredmanFibonacciHeapsTheir1987}, 
\textsc{DIJKSTRA} can achieve a time complexity of O($m$ + $n\log n$) (where $n$ and $m$ are the number of nodes and edges respectively).

\subsection{Limitations}
\label{subsec:limits}
Recently, Duan et al. in their famous paper \textit{Breaking the Sorting Barrier for Directed Single-Source Shortest Paths}\cite{duanBreakingSortingBarrier2025}, 
improved the time complexity of \textsc{DIJKSTRA} to $O(m \log^{2/3}n)$, 
yet \textsc{DIJKSTRA} still has a fundamental flaw, it's working "blind" without knowing whether or not if the path it is exploring will lead towards a target node $t \in T$. 

Specifically in our case, on the SSMTSP problem, \textsc{DIJKSTRA} still falls shorts in many cases, leading to dead-ends, as it is forced to explore every node with a smaller distance cost to a target node, even if its inefficient and unnecessary to do so.

This inefficiency highlights the need of algorithms which use external information.
Heuristic algorithms like A*\cite{hartFormalBasisHeuristic1968b} attempt to combat this issue, but even then, we require specific knowledge to calculate the heuristic value $h(n)$, 
on which we have the inability to dynamically re-calculate computationally all the time for multiple targets.

\textsc{DIJKSTRA-PREDICTION}, by utilizing machine learning techniques to prune the search space, manages to skip irrelevant edges, while still maintaining a guarantee of the optimal shortest path, but yet still falls short of tackling this issue.

\section{Comparative Analysis of Other Algorithms}
\label{sec:compare_dijkstra}

To fully understand the trade-offs of \texttt{DIJKSTRA}, it is essential to evaluate it against alternative path-finding algorithms.

Compared to other algorithms, \texttt{DIJKSTRA} maintains its place as a foundational \textit{stepping stone} due to the nature of its algorithmic design. Specifically, \texttt{DIJKSTRA} offers a unique balance of determinism, exactness, and flexibility, which we will further explore.


\subsection{Dynamic Programming: The Bellman-Ford Algorithm}
\label{subsec:compare_dijkstra}

A downside of \texttt{DIJKSTRA}, is its reliance on non-negative weights ($w(e) \geq 0$) to find the shortest path. If we try solving a graph with any negative weights ($\exists e \in E \text{ s.t. } w(e) < 0$), the greedy nature of \texttt{DIJKSTRA} causes it to fail, because it may cause a premature finish of the algorithm, not allowing it to later find a cost-reducing path through a negative edge. 

To showcase this failure, consider the simple graph in Figure \ref{fig:negative_edge_example} with three vertices. The edges are defined as $(A, B)$ with $w=5$, $(A, C)$ with $w=2$, and a negative edge $(B, C)$ with $w=-10$ %.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\linewidth]{tex/figures/figure_dijkstra_vs_bellman.pdf}
    \caption{A simple directed graph featuring a negative edge weight. \newline \textsc{DIJKSTRA}'s greedy approach incorrectly settles node $C$ before discovering the negative edge $(B, C)$.}
    \label{fig:negative_edge_example}
\end{figure}

When standard \textsc{DIJKSTRA} processes this graph, it executes the following sequence:
\begin{itemize}
    \item \textbf{Initialization:} Distance to $A = 0$, while $B = \infty$ and $C = \infty$ %.
    \item Step 1 (Extract A) The algorithm extracts $A$ (the minimum distance node). It updates its neighbors, setting $B = 5$ and $C = 2$ %.
    \item Step 2 (Extract C): The algorithm extracts the next minimum distance node, which is $C$ (since $2 < 5$). Because $C$ has no outgoing edges, it is marked as visited. \textsc{DIJKSTRA} now irrevocably locks in the shortest path to $C$ as $2$ %.
    \item Step 3 (Extract B): The remaining node $B$ (distance $= 5$) is extracted. The algorithm relaxes edge $B \rightarrow C$. The cost to reach $C$ through $B$ would be $5 + (-10) = -5$ %.
\end{itemize}

At finalization \texttt{DIJKSTRA} claims the shortest path to $C$ is $2$ (via $A \rightarrow C$), in reality we can achieve a distance of $-5$ (via $A \rightarrow B \rightarrow C$).

To tackle this issue, Bellman-Ford\footnote{Although often referred to as the Bellman-Ford algorithm, this method was developed and published independently by Alfonso Shimbel (1955), Lester Ford Jr. (1956), Edward F. Moore (1957), and Richard Bellman (1958). It is often also referred to as the Bellman-Ford-Moore algorithm.}\cite{bellman:routing} uses a dynamic approach. Unlike the greedy approach of extracting the edge with the current least weight/cost, it relaxes all $m$ edges across $n-1$  iterations, resulting in an time complexity of $\mathcal{O}(nm)$. Thus, although Bellman-Ford offers a \textit{heavier} computational cost compared to that of \texttt{DIJKSTRA} ($O(m+n\log n$), it provides a solution to the negative edge problem. 

However, even so in real-life application, where edges usually represent measurements of distance or time, edges are inherently non-negative. It is for that fact, that the greedy approach \texttt{DIJKSTRA} has to offer can usually be a far better option when it comes to computational efficiency.

\subsection{Heuristic and Goal-Directed Search: A* and ALT}
\label{subsec:alphastar_alt}
As we previously discussed on Section \ref{subsec:limits}, \texttt{DIJKSTRA}'s greedy expansion causes the unnecessary consideration and exploration of irrelevant \textit{dead-end} nodes if their accumulated cost is smaller compared to the accumulated cost of a target node. Figure \ref{fig:blind_limit} visually showcases this flaw.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.8\linewidth]{tex/figures/figure_blind_limit.pdf}
    \caption{To reach the target $T$ (total cost 20), \texttt{DIJKSTRA}'s greedy expansion forces it to first explore the irrelevant nodes $N_1, N_2,$ and $N_3$, simply because their accumulated cost (5) is locally smaller.}
    \label{fig:blind_limit}
\end{figure}

To combat this exact limitation, heuristic and goal-driven algorithms utilize external information on directional orientation data to significantly reduce the search space and avoid irrelevant nodes.
\newpage

The A* algorithm\cite{hartFormalBasisHeuristic1968b} augments \textsc{DIJKSTRA} by incorporating a heuristic function $h(u)$, which estimates the remaining cost from an active node $u$ to a specific target node $t$. By sorting the priority queue based on the evaluation function $f(u) = g(u) + h(u)$, where $g(u)$ represents the exact accumulated cost from the source, A* \textit{pushes} the frontier expansion toward the destination.

The ALT\cite{goldbergComputingShortestPath2005} algorithm, refines this approach by combining A* search with Landmarks and the Triangle Inequality  $d(u,v) \ge |d(l,v) - d(l,u)|$ to derive lower bounds without requiring geometric coordinates. 

While these approaches are drastically faster to \texttt{DIJKSTRA} in point-to-point (P2P), they suffer from a fundamental flaw of their own.  In the SSMTSP framework, which Feijen and Schäfer tackle, the execution stops as soon as the algorithm settles the \textit{closest} target $t \in T$. Designing and configuring a heuristic $h(u)$ for this problem introduces a huge bottleneck. At initialization, the algorithm does not know which of the targets $t \in T$ is actually the closest, and as a result a goal-directed heuristic must continuously evaluate the distance to all potential targets to avoid overestimation (e.g., computing $h(u) = \min_{t \in T} h_t(u)$)
Often, performing this evaluation for larger target sets will lead the algorithm to higher computational cost, compared to what it saves from pruning. Therefore, \texttt{DIJKSTRA}'s uniform blind search is quite often preferred, as it naturally stops at the nearest target node $t \in T$, without requiring any \textit{heavy} evaluation. 


\subsection{Pre-processing Approaches: Contraction Hierarchies}
\label{subsec:preprocessing_schemes}

In web applications, classical algorithms are often \textit{put aside} in favor of advanced pre-computation schemes, such as Contraction Hierarchies\cite{geisbergerContractionHierarchiesFaster2008}. These techniques execute an expensive offline pre-processing phase to insert shortcut edges (See Figure \ref{fig:shortcut_edge}). For instance, in a Contraction Hierarchy, if the optimal route from node $A$ to node $C$ strictly passes through a lower-importance node $B$, the algorithm \textit{contracts} $B$ and inserts an artificial shortcut edge $A \rightarrow C$ with a combined weight of $w(A,B) + w(B,C)$. By pre-computing millions of such shortcuts, online algorithm can ignore dense local networks resolve shortest paths in microseconds.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.8\linewidth]{tex/figures/figure_shortcut_edge.pdf}
    \caption{A visual representation of a Contraction Hierarchy operation. Node $B$ is contracted, and a new shortcut edge $A \rightarrow C$ is inserted.}
    \label{fig:shortcut_edge}
\end{figure}

Despite their unmatched query speed, these do not eliminate the need of classical algorithms like \texttt{DIJKSTRA}. First, all hierarchical pre-computation engines fundamentally rely on standard \textsc{DIJKSTRA} as a mechanism to execute the initial offline graph contractions of the pre-processing phase. Second, these modern methods collapse when it comes to dynamic graphs. For example, in a GPS system, if a sudden traffic accident drastically increases the traversal time (weight) of a major highway link (edge), the pre-computed shortcuts that utilize that link become instantly invalid, after which the systems \textit{old} hierarchy becomes effectively useless. The only solution then would require a computationally expensive re-calculation phase of the affected network regions, in order to remake any possible shortcuts.

Classical \textsc{DIJKSTRA} does not require any geometric coordinates, external spatial data, or structural pre-computation to guarantee an exact global optimum. Its simplicity, effectiveness, and capacity to handle dynamic edge-weight variations explain why it remains an important operational algorithm inside modern routing, telecommunication, and networks.
